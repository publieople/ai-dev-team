## 数据流

### 初始化流程

```
1. 主Agent读取项目根目录
   ↓
2. 扫描文件结构 → 识别技术栈
   ↓
3. 生成 project-analysis.json
   ↓
4. 生成《项目理解报告》(Markdown)
   ↓
5. 等待人类确认
   ↓
6. 创建 .ai-dev-team/ 目录结构
   - config.json
   - state.json
   - tasks/ 目录
   - reports/ 目录
```

### 开发循环

```
1. 主Agent分析项目 → 识别改进点
   ↓
2. 生成任务规划 → 写入 state.json
   ↓
3. [PENDING_APPROVAL] 生成规划报告
   ↓
4. 人类确认（或auto_approve）
   ↓
5. [ASSIGNED] 创建子Agent
   ↓
6. 子Agent读取任务卡片 + 代码上下文
   ↓
7. [IN_PROGRESS] 子Agent执行
   ↓
8. 子Agent生成执行报告
   ↓
9. [TESTING] 主Agent验证（测试/检查）
   ↓
10. [PENDING_HUMAN_TEST] 生成验收报告
   ↓
11. 人类验收
   ↓
12. [COMPLETED] Git提交
   ↓
13. 子Agent销毁
```

## 关键设计决策

### 1. 单任务单Agent

**原因：**
- 简化状态管理
- 避免Agent间竞争
- 清晰的职责边界
- 便于成本控制

**实现：**
```python
# 伪代码
active_agent = None

def assign_task(task):
    if active_agent:
        queue.append(task)
        return
    
    active_agent = spawn_agent(task.type)
    active_agent.run(task)
    
def on_agent_complete(agent, report):
    active_agent = None
    process_report(report)
    if queue:
        assign_task(queue.pop(0))
```

### 2. 状态机持久化

**原因：**
- 主Agent可重启恢复
- 人类可随时查看进度
- 支持审计追溯

**实现：**
- 每次状态变更 → 写入 state.json
- 使用TOON格式减少token

### 3. Git可追溯

**原因：**
- 不干扰人类开发者
- 完整审计链路
- 便于回滚分析

**实现：**
- 提交信息嵌入元数据
- 单独分支或标记

### 4. 上报机制

**分级上报：**
```
子Agent失败
  → 重试3次
    → 上报主Agent
      → 主Agent分析
        → 能解决 → 重新指派
        → 不能解决 → 生成问题报告 → 上报人类
```
